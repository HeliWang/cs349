Overview

The goal of this assignment is to build an interactive game, demonstrating your mastery of:

  • Model-View-Controller design patterns
  • Java Swing, 2D graphics and transformations
  • Direct manipulation and using a mouse to interact with shapes

You will build a real-time game, similar to Fruit Ninja. In this game, shapes move onto the screen from random positions at the bottom of the screen. The player must use the mouse to “slice” fruit into two slices; points are awarded for every piece of fruit that is successfully split.

Requirements

The following guidelines describe gameplay:

  • Your game should consist of a window, displaying the title, high-score, elapsed time and play area. Your game should be entirely playable using a mouse.
  • When the game is running, random fruit shapes will fly up from the bottom of the screen, reach the top of their ascent somewhere in the middle of the screen, and then drop and fall off the bottom of the screen.
  • The player interacts with fruit by slicing it: pressing the mouse button, dragging across a piece of fruit, and then releasing the mouse button. The slice created in the fruit should match up with the line created by the start/end points of the mouse drag. The alignment doesn’t need to be perfect, but it should be a reasonable match (so that you can, for instance, bisect a piece of fruit into smaller 1/5 and 4/5 size pieces).
  • After a piece of fruit is split, it must break apart as described above, and the remaining two pieces should then fall off the screen.
  • You don’t need to allow sub-slices (i.e. you don’t need to support splitting fruit into more than two pieces).
  • Points are awarded for each piece of fruit that is successfully sliced. A high-score should display a running total. You can choose a meaningful scale.
  • A timer should indicate how much time (min:sec) has elapsed for the current game.
  • If more than 5 pieces of fruit are missed (i.e. allowed to fall off the bottom of the screen without being sliced), then the game ends. When this happens, you should allow the user to start a new game (which clears the high score and resets the timer!)
  • Your game should be compelling: animations should happen at a fast-enough pace that an average player or TA can successfully play the game.
  • You must support more than a single piece of fruit on-screen at the same time: you can have a random number of shapes animating, or have a progression (i.e. it starts out easy and adds more pieces as you continue playing). You need to decide what is compelling.

The following are implementation requirements:

  • Your game just be built entirely in Java 7 (1.7.025 is installed in the VM and that’s fine). No third-party toolkits or libraries are allowed (see below) but you may use any of the sample code provided in this course.
  • Starter code is provided (link below), which you can use as the basis of your assignment. The starter code is an almost-complete implementation of drawing and slicing fruit: filling in the missing pieces of the Fruit class will make it functional. Note that doing this will not get you a complete application; you will still need to add animation, timers and so on.
  • You can make any changes that you want to the starter code (or even write everything yourself), except that you must have a Fruit class that implements FruitInterface.
  • The UI should be drawn using Java Swing components. The sample code is fairly functional, and uses a java.awt.Area to represent Fruit. This is particularly helpful, as it provides intersection methods that you can use when slicing your fruit. If you want more complex shapes (e.g. using bitmaps for textures), you will probably need to write a custom class to do this - just make sure to implement the FruitInterface.
  • Your code must utilize the MVC pattern, and your interface should consist of one or more views. The model should contain the list of shapes to be drawn. The MVC implementation in the sample code is a reasonable starting point.
  • This is a real-time game, and should exhibit smooth animations and gameplay. Marks will be partially based on performance and interactivity.
  • As per the submission guidelines below, you must include a makefile.

Download starter code here. To install the starter code in SVN:

  1. svn update your SVN working directory.
  2. mkdir a3 to create your assignment directory.
  3. Download the starter code and copy the zip file into this directory.
  4. unzip a3starter.zip to unzip the file.
  5. svn add a3/ to add the directory and all of its contents, then svn commit everything.
  6. svn status should not show any changes pending.

Testing

You do not need to write unit tests for this assignment. Note however, that TAs will aggressively deduct marks if your assignment crashes, segfaults, or otherwise behaves erratically. Test test test!

Submission

All code must be checked into SVN by the deadline specified above. You must provide a makefile that will build and run your application. You may check in your IDE project files but we will not open and build your project in an IDE. Your project must compile and run from the command line to receive full marks. This means that

  • make will build your project
  • make run will build and run it
  • make clean will remove all *.class files

Also, when submitting in SVN, remember to

  • svn add all of your files that you want to include
  • svn commit all of your changes

To check that you’ve submitted properly, svn checkout to a different directory and make sure that builds and works properly. If your second checkout works differently, it’s because you’ve made a mistake submitting it.

Getting Started

Here’s some suggestions on how to get started.

  • Unzip the starter code into SVN, as described above.
  • Sketch out how you want your application to work, and how you need to support that functionality (i.e. how your model should behave, what views you might need).
  • Read the sample code, and modify the Main, View and Model to suit your functionality.
  • Iterate to progressively add features. Build the simplest thing that works, then iterate on that.
  • Follow the hints provided in the starter code to get it working.
  • Check into SVN often! We only look at the final submission, so there’s no harm in checking in code as you’re building it.
  • Use an IDE! There are some excellent Java tools that will greatly improve your productivity, and make it easier to debug problems that will crop up. Eclipse and IntelliJ are both excellent Java IDEs and highly recommended. (Note: the VM doesn’t have an IDE installed by default, but it’s simple to download and install from the sites listed above).

Non-Requirements

The following are things that you should not do for this assignment.

  • Do not include any third-party libraries. Your code should rely exclusively on native Java packages. An exception is the use of third-party resources (art, music), which may be used if their license allows it (e.g. Creative Commons license).
  • The Google logging framework glog that you used in A1 and A2 isn’t required (or available in Java). The java.util.logging package is recommended for logging and debugging.
